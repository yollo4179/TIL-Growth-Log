
# 근본적인 문제 : 

1. 근본적인 문제: 웹(HTTP)은 기억 상실증에 걸려 있다.
우리가 인터넷을 할 때 사용하는 `HTTP 프로토콜`은 `**무상태성(Stateless)**` 이라는 특징을 가지고 있습니다.<br>
서버는 `클라이언트`(브라우저)의 `이전 요청`을 전혀 `기억하지 못합니다`<br>
방금 로그인을 했어도, 1초 뒤에 "내 프로필 보여줘"라고 `새로 요청`하면 서버는 `"누구세요?"`라고 되묻습니다.<br>

`이전에 로그인한 바로 그 사람`이다"라는 `상태를 유지(Stateful)`하기 위해 우리는 `쿠키와 세션`이라는 기술을 사용합니다.<br>


# Stateful한 상태 유지하기,  인증(Authentication)과 인가(Authorization)

## 인증 (Authentication - '넌 누구냐?'):
- 사용자의 `신원을 확인하는 과정`입니다.
- "내 아이디는 dev_king이고, 비밀번호는 1234야." -> 서버: "오, DB에 있는 정보랑 일치하네. 너 맞구나!" (로그인)
- 사실 `jwt를 사용`하는 겨우, 쿠키에 `Refresh 토큰`이 있다면, 매번 로그인하지 않아도  서버에서 access 토큰을 얻어와 로그인을 유지할 수 있습니다.

## 인가 (Authorization - '이거 할 권한이 있느냐?'):

- 인증된 사용자가 `특정 리소스`나 `서비스 및 기능`에 접근할 수 있는 권한이 있는지 확인하는 과정입니다.

- 나 방금 로그인한 dev_king인데, 다른 사람 글 좀 지워볼게." -> 서버: "넌 로그인(인증)은 됐지만 일반 유저라서 관리자 권한(인가)이 없어. 접근 금지!"


# 🍪 쿠키 (Cookie)

## `정의:` `브라우저`에 저장되는 아주 작은 텍스트 데이터입니다.

특징 (가장 중요!): 브라우저는 특정 서버(도메인)에 `요청을 보낼 때마다`, 해당 서버에서 예전에 구워준(저장해 둔) `쿠키를 HTTP 요청 헤더`에 알아서(`자동으로`) 딱 붙여서 `보냅니다.`<br>

`단점`: `브라우저에 텍스트 형태로 적나라하게 저장`되므로, 여기에 비밀번호 같은 걸 넣으면 해커들에게 "내 정보 가져가세요" 하는 것과 같습니다.<br>

# 🎫 세션 (Session)
정의: `사용자의 중요한 정보`나 상태를 `서버의 메모리나 데이터베이스`에 안전하게 `저장`하는 방식입니다.<br>

동작 방식: `서버에 진짜 정보를 숨겨`두고, 클라이언트에게는 그 `정보에 접근할 수 있는` key(무의미하고 복잡한 문자열)을 하나 발급해 줍니다. 이것이 바로 `**세션 ID (세션 토큰)**`입니다.<br>

`놀이공원 VIP 팔찌`라고 생각하시면 됩니다.<br>

## 인증 및 인가 과정 ( 프론트와 백엔드에서 일어나는 전체 흐름)
```html

이제 유저가 브라우저를 열고 로그인을 할 때 어떤 일이 벌어지는지 시간 순서대로 엮어보겠습니다.

1. [프론트] 로그인 요청 (인증 시작): 사용자가 ID/PW를 치고 로그인 버튼을 누릅니다.

2. [백엔드] 인증 및 세션 생성: 서버가 DB를 확인해서 정보가 맞으면 로그인을 승인합니다. 그리고 서버의 저장소에 [유저 A의 정보]를 기록하고, 그에 매칭되는 Session ID: sdf8923jkfds를 생성합니다.

3. [백엔드 -> 프론트] 쿠키 굽기: 서버는 프론트엔드에 응답을 줄 때, HTTP 응답 헤더의 Set-Cookie라는 항목에 방금 만든 Session ID를 담아서 보냅니다.

4. [브라우저] 쿠키 저장: 브라우저가 이 응답을 받으면, "아, 서버가 쿠키 저장하래!" 하고 자기만의 쿠키 보관함에 Session ID를 저장해 둡니다.

5. [프론트 -> 백엔드] 인가 요청: 유저가 "내 장바구니 보기" 페이지를 클릭합니다. 이때 브라우저가 똑똑하게 아까 저장해둔 Session ID 쿠키를 HTTP 요청에 자동으로 같이 딸려 보냅니다.

6. [백엔드] 인가 확인: 서버는 요청을 받으면 쿠키 안에 있는 Session ID를 봅니다. "어? 이 팔찌 번호(Session ID) 우리 서버에 있나? 아, 아까 로그인한 유저 A의 팔찌네!" 하고 유저를 식별한 뒤, 장바구니 데이터를 프론트엔드로 보내줍니다.


```

## 세션의 문제점  

질문 : 세션은 서비스 사용할때마다 인가과정을거치니까 db를 뒤지지?<br>
AI : 사용자가 클릭을 하거나 페이지를 이동할 때마다 브라우저는 세션 ID(쿠키)를 서버로 보내고, 서버는 "이 세션 ID가 유효한지, 이 유저가 누구인지, 권한은 있는지" 확인하기 위해 저장소를 뒤져야 합니다.<br>

## 무거운 DB(디스크) 대신 `Redis 또는 Memcached`를 사용할 수도 있지만..... 
```html
`메인 DB` (디스크 기반): 회원 정보, 결제 내역 등 절대 날아가면 안 되는 `영구적인 데이터 저장.` (느림)

`Redis` (메모리 기반): 세션 ID처럼 빠르게 읽고 써야 하며, 날아가도 (다시 로그인하면 되니까) `치명적이지 않은 임시 데이터 저장.` (엄청나게 빠름)

조회를 하긴 하지만, `디스크가 아닌 RAM(Redis)을 뒤지기 때문`에 눈 깜짝할 사이에 인가 과정이 처리되는 것입니다.
```

# 아예 안 뒤지면 안 돼?(JWT의 등장)
- 세션: 쿠키에는 '열쇠(Session ID)'만 있고, 진짜 정보는 서버가 가지고 있음. (저장소 조회 필수)

- JWT (토큰 방식): 토큰 자체에 `유저의 정보와 권한`을 `암호화`해서 다 때려 넣음. 서버는 요청이 오면 DB나 Redis를 뒤질 필요 없이 `암호화 알고리즘(CPU 연산)`으로 이 토큰이 `위조되지 않았는지만 검사(디코딩)`하면 끝입니다.

## `짚고 넘어가야할 교정`: <br>
JWT의 앞부분은 `**'암호화'된 것이 아니라 단순한 'Base64Url 인코딩'**`이라는 점입니다.<br>
누구나 디코딩(해독)해서 안의 내용을 제이슨(JSON) 형태로 볼 수 있습니다.<br>
그럼 왜 안전하냐고요? 말씀하신 `'해시(Hash)'`를 이용한 `'서명(Signature)' 검증` 때문입니다.<br>

JWT 토큰을 실제로 보면 `.` (점)을 기준으로 `세 덩어리`로 나뉘어 있습니다
```html

1) 헤더 (Header - xxxxx): 
토큰의 타입(JWT)과 서명을 만들 때 쓸 `alg:해시 알고리즘`(예: HS256, RS256) 정보가 들어있습니다. (Base64Url 인코딩됨)

2) 페이로드 (Payload - yyyyy):
여기가 바로 말씀하신 `특정 유저 식별 정보`가 제이슨(JSON)으로 들어가는 곳입니다.
유저 ID, 권한(역할), 토큰 만료 시간 등이 담깁니다. (Base64Url 인코딩됨)

3) 서명 (Signature - zzzzz): 
여기가 JWT의 심장입니다. 
서버만 알고 있는 **'비밀키(Secret Key)'**를 사용해서, 헤더와 페이로드를 인자로 `지정된 해시 알고리즘으로 돌려` 만든 결과물입니다.


`조작 불가` 
-- 개발자 모드에서 base 64Url로  디코딩하여 권한 및 페이로드 정보를 수정할 수 있지만, 서버의 비밀키를 모르면 
절대!!! 서버에서 `인가하는 과정`에서 해시값의 결과가 `서명의 값과 일치하지 않기 때문에` 절대 서비스에 접근할 수 없습니다. 
```
JWT를 이용해 인증 및 인가 과정이 서버에서 실제로 돌아가는 순서 (동작 원리)
```html

1. Base64 디코딩 및 제이슨 변환 (유저 식별):
서버는 토큰을 받으면 중간 부분(Payload)을 Base64로 슥 디코딩해서 JSON으로 엽니다.

* "오, 열어보니까 user_id: 123, role: admin 이라고 적혀 있네?" (유저 식별 완료)


2. 해시를 통한 서명 검증 (위조 확인 - 이 부분이 핵심!):

하지만 서버는 이 정보를 바로 믿지 않습니다. 해커가 페이로드를 조작했을 수도 있거든요. 그래서 서버는 자기가 가진 **'비밀키(Secret Key)'**를 꺼내서, 방금 받은 헤더와 페이로드를 직접 해시 함수에 넣어 돌려봅니다.

- "내가 직접 해시 돌려본 결과값(A)이랑, 토큰 맨 끝에 붙어온 서명값(B)이 똑같네? 중간에 아무도 안 건드렸구나!" (검증 통과)

3. 인가 (Authorization):
검증이 끝났으니 비로소 권한을 부여합니다.

"위조되지 않은 진짜 토큰이고, role이 admin이니까 관리자 페이지 접속을 허락(인가)해 줄게!"
```
## 요약

JWT란, Base64로 인코딩된 토큰을 디코딩하여 JSON으로 유저를 식별하고, 서버의 비밀키를 이용한 해시(서명) 검증을 통해 데이터가 조작되지 않았음을 확인한 뒤 인가 처리를 해주는 방식이야!<br>
